## Context

Material-label calibration analysis found that processed .chesseval files contain syntactically parsed but semantically impossible chess positions. In 1,000,000-row deterministic prefix samples, invalid piece geometry appeared in train, validation, and test at roughly 0.04%, including rows with impossible material differences such as -479..523 and examples like dozens of queens on one side. The current Rust preprocess path accepts Board::from_str rows but does not enforce strict standard-position constraints.

## Why It Is Needed

Invalid board geometry corrupts material-difference features, fixed material-prior experiments, calibration analysis, and model training. Even if rare, extreme rows can create very large logits, gradient spikes, and misleading evaluation signals. Dataset regeneration should happen before further serious model experiments.

## Completion Criteria

Preprocessing rejects or quarantines non-standard positions with strict, documented constraints. Regenerated train/validation/test processed datasets have validity reports showing zero invalid piece geometry under the analyzer. The material-label calibration analysis is rerun on the regenerated data, and docs record the old-data contamination rate, new-data validity, and training implications.

## Detailed Plan

1. Define strict standard-position constraints for processed data. 2. Analyze raw `data/raw/lichess_db_eval.jsonl` and DuckDB `rows` samples to identify the FEN patterns that produce impossible processed positions. 3. Add Rust preprocess validation before writing rows. 4. Count rejected rows by reason and print a preprocessing report. 5. Add Rust tests for invalid piece counts, missing kings, pawn-count violations, overlap/impossible occupancy if representable, and valid promoted-piece edge cases. 6. Regenerate processed datasets. 7. Rerun material-label calibration analysis on regenerated splits. 8. Decide whether to discard existing checkpoints trained on contaminated data or keep them only as historical baselines.
