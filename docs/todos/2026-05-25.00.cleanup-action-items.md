## Context

This TODO tracks cleanup work found during the initial project review on
2026-05-25. The project mixes Python model/training/evaluation code with Rust
preprocessing, ONNX inference, and benchmarking code. The review found several
artifact-compatibility, correctness, packaging, documentation, and repository
hygiene issues.

Priority guide:

- P0: Breaks a primary workflow or can produce clearly wrong chess/eval behavior.
- P1: Blocks maintainability, packaging, reproducibility, or common developer use.
- P2: Nice-to-have cleanup that reduces confusion or future drift.

## Why It Is Needed

The cleanup work needs a canonical tracker because the project had drift between
current code, committed model artifacts, README guidance, and utility scripts.
Some P0 issues have already been fixed, but follow-up P1/P2 items remain and
should be handled deliberately instead of rediscovered repeatedly.

### P1: Clarify Material Feature Semantics

- [x] Decide whether the material feature should be computed from the original
      board tensor or from learned trunk activations.
- [x] If board-material is intended, pass the original input tensor or a
      precomputed `material_diff` into `ValueHead`.
- [x] Reject the learned-activation material behavior and keep the comments and
      arguments aligned with explicit board material.

Decision:

- The material feature is intended to be explicit board material, not a learned
  trunk activation slice.
- `ValueOnlyModel` and `ModelFull` should derive raw material difference from
  the original `[B, 20, 8, 8]` board tensor before coordinate/trunk layers and
  pass it into `ValueHead`.
- `ValueHead` may consume trunk activations for its convolutional path, but must
  not infer material from those trunk activations.

Implementation slice pseudo diff:

```diff
~ python/libs/model.py
+ def _material_diff_from_board(board: Tensor, material_weights: Tensor) -> Tensor:
+     validate board tensor shape is [B, 20, 8, 8]
+     return weighted sum(board[:, 6:12]) - weighted sum(board[:, 12:18])
~ def _material_feature(x, material_weights, material_scale, material_diff=None):
+     if material_diff is None:
+         material_diff = _material_diff_from_board(x, material_weights)
+     return tanh(material_diff / alpha).unsqueeze(1)
~ class ModelFull.forward / forward_value:
+     raw_material_diff = explicit argument or _material_diff_from_board(original_input)
+     value = self.value_head(trunk_output, raw_material_diff)
~ class ValueOnlyModel.forward:
+     raw_material_diff = explicit argument or _material_diff_from_board(original_input)
+     return self.value_head(trunk_output, raw_material_diff)

+ python/tests/test_model_material.py
+ assert raw material diff for known board tensor
+ assert model passes board-derived material diff into ValueHead
+ assert ValueHead rejects trunk activation material inference without explicit material_diff
```

Evidence:

- `_material_feature` documents an input shaped `[B, 20, 8, 8]`.
- `ValueOnlyModel.forward` currently passes trunk output shaped
  `[B, CHANNELS, 8, 8]` into `ValueHead`.
- A sample board with our pawn vs their rook should produce
  `tanh((1 - 5) / 5) = -0.664...`, but the previous trunk-activation path
  produced an unrelated near-zero value.
- Local wandb summaries/configs were not available in the checkout, so the
  historical experiment uplift could not be verified locally.

Acceptance checks:

- Comments match implementation.
- A small unit test verifies material-diff behavior for a known board tensor.
- Existing checkpoints still load, but their outputs intentionally change
  because material is now computed from board state instead of trunk channels.
