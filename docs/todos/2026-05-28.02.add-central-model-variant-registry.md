## Context

Model variant identifiers are duplicated across python/libs/model.py, python/cli.py help text, training validation, tests, benchmark defaults, and experiment logs. Variant metadata such as family, status, material handling, expected parameter count, and intended use is implicit.

## Why It Is Needed

A central registry will make active, baseline, experimental, and dormant variants explicit and prevent CLI/help/benchmark drift as research variants accumulate.

## Completion Criteria

A registry defines all supported model variants and metadata. Training CLI validation, help text, model factory dispatch, benchmark candidate selection, and tests consume the registry instead of duplicating variant strings. Existing variant IDs and checkpoint loading behavior remain unchanged.

## Detailed Plan

1. Define ModelVariantSpec with id, family, status, description, material behavior, expected params, and default inclusion flags.
2. Populate specs for all current variants.
3. Route CLI validation and allowed-value text through the registry.
4. Route benchmark default candidate metadata through a separate checkpoint candidate list linked to variant specs where appropriate.
5. Add tests that registry IDs match supported model construction and checkpoint variant resolution.
