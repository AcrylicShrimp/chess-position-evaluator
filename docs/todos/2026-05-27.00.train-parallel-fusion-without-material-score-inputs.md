## Context

The clean-data parallel fusion diagnostic suggests the MaterialFeatureValueHead may contribute to overconfident failures in sharp tactical/endgame cases. The next controlled architecture test should remove explicit material-score-derived inputs entirely, not merely weaken them.

The later `parallel-cnn-attn-aligned-add` experiment tested a cleaner branch-fusion arithmetic while keeping `MaterialFeatureValueHead`; it finished in the same performance class as the previous concat-fusion material-feature run and was slightly worse on full validation/test BCE. This reduces the likelihood that branch scale/basis alignment alone explains the parallel-fusion gap.

## Why It Is Needed

Validation-prefix diagnostics showed the parallel model was especially worse in confident-label and moderate material-imbalance buckets. Source tracing later showed that material/label sign conflict is present but not dominant, so removing material inputs is an isolation experiment rather than a proven fix.

Because aligned-add did not improve the material-feature parallel run, the next highest-value controlled variable is the value-head input contract. The experiment should use the simpler existing `parallel-cnn-attn-fuse` trunk, not aligned-add, so the only intended behavioral change is removing explicit material-score-derived inputs.

## Completion Criteria

A new model variant removes material-score-derived inputs from the parallel-fusion value head, trains on the cleaned dataset with the established recipe, and is evaluated against the clean stacked edge-gate FFN and previous parallel material-features run.

## Detailed Plan

1. Add a `parallel-cnn-attn-fuse-no-material` variant using the existing `ParallelCnnAttentionTrunk`.
2. Route that variant to `ResidualValueHead`, so the value head receives only learned board/trunk features and no explicit material-diff feature.
3. Preserve the cleaned dataset and established warmup-cosine recipe exactly: 100 epochs, 1024 steps, batch 2048, lr 0.0005, wd 0.0001, warmup 5, eta-min 1e-6, grad clip 5.
4. Run unit tests covering model construction, forward path, CLI variant exposure, and material-diff independence for the new variant.
5. Run a target-batch CUDA smoke training gate.
6. Run the full training experiment only if smoke and tests pass.
7. Evaluate full validation/test splits and compare primarily against the clean `parallel-cnn-attn-fuse` material-feature run, secondarily against the clean stacked edge-gate FFN baseline.
8. Record the full result under `docs/experiments` and close this TODO.
