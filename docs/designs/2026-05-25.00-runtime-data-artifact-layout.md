# Runtime Data And Artifact Layout

## Goal

Keep source control focused on code, tests, and documentation while giving local
runtime data and model artifacts predictable filesystem locations.

## Agreed Model

Runtime data lives under `data/`:

```text
data/
├── raw/
├── interim/
└── processed/
```

Runtime model artifacts live under `artifacts/`:

```text
artifacts/
├── checkpoints/
└── onnx/
```

Only README and `.gitkeep` files are committed from these directories. Actual
datasets, source archives, staging databases, checkpoints, and ONNX exports are
ignored local files.

## Boundaries And Invariants

- `data/raw/` stores downloaded source files such as `lichess_db_eval.jsonl`.
- `data/interim/` stores local staging files such as DuckDB temporary databases.
- `data/processed/` stores generated training and validation datasets.
- `artifacts/checkpoints/` stores local training checkpoints.
- `artifacts/onnx/` stores local ONNX exports and optional external data files.
- No `.pth`, `.onnx`, `.onnx.data`, `.jsonl`, `.chesseval`, `.duckdb`, or
  `.duckdb.tmp` files should be committed.
- Git LFS is not required unless a separate explicit migration is chosen later.

## Tradeoffs

Keeping artifacts out of git makes fresh clones smaller and avoids treating
experiment outputs as source. The tradeoff is that inference, export, benchmark,
and analysis commands require local artifact preparation before use.

Existing model artifacts are preserved locally under `artifacts/`, but removed
from source control.

## Rejected Alternatives

- Keeping committed model artifacts under `models/`: convenient but makes source
  control own experiment outputs.
- Expanding Git LFS immediately: unnecessary while the project has no committed
  LFS workflow and `git lfs` is not part of the local tooling.
- Leaving raw data in crate directories or repository root: unclear ownership
  and risky with 100 GB scale data files.
