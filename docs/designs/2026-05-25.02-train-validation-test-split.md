# Train Validation Test Split

## Design Goal

Create a clean dataset split contract for future experiments so model selection and final evaluation use separate data. This is a hard cutover from the current two-way split to a three-way split.

## Current State

- Preprocess writes `train.chesseval` and `validation.chesseval`.
- The split ratio is effectively 90% train and 10% validation.
- The evaluator can report aggregate metrics for `train` and `validation`.
- There is no independent test split, so validation metrics are useful for model selection but weak as final performance claims.
- Existing checkpoints were trained under the old split contract and should not be used as clean baselines for the new final test metric.

## Agreed Model

Use a three-way processed dataset contract:

```text
data/processed/train.chesseval
data/processed/validation.chesseval
data/processed/test.chesseval
```

The split ratio is:

```text
train:      90%
validation: 5%
test:       5%
```

The split is contiguous after the existing randomized DuckDB `rows` table is created:

```text
train      = rows[0..train_size)
validation = rows[train_size..train_size + validation_size)
test       = rows[train_size + validation_size..dataset_size)
```

`test_size` is computed as the remaining rows after train and validation sizes are computed. This preserves total row count even when integer truncation is involved.

The source windows are computed before FEN parsing. Invalid FEN rows are skipped
when `.chesseval` files are written, so preprocess must report the actual
written row count for each split. The existing randomized `rows` table keeps the
post-skip split ratios stable enough for this fixed-ratio contract.

## Boundaries And Invariants

- `train.chesseval` is used for optimizer updates.
- `validation.chesseval` is used for model selection and best-checkpoint decisions.
- `test.chesseval` is used only for final evaluation and model comparison after training decisions are made.
- `cpe train` should continue to load only train and validation files.
- `cpe eval-dataset --split test` should resolve to `data/processed/test.chesseval`.
- Validation reports should continue warning that validation is model-selection data.
- Test reports should not carry the validation warning.
- No compatibility shim is needed for the old 90/10 split. Missing `test.chesseval` should fail clearly when `--split test` is requested.
- Generated `.chesseval` files remain ignored by Git.
- Preprocess output should distinguish source-window sizes from actual written row counts.

## Failure Modes

- Preprocess should remove stale `train`, `validation`, and `test` outputs before writing new split files.
- If preprocess is run with old generated data present, all three processed split files should be regenerated together.
- If the user evaluates `--split test` before regenerating data, the command should fail with a clear missing-file message.
- Old checkpoints may still be loadable, but their test scores must be treated as transitional diagnostics rather than clean final baselines.

## Tradeoffs

- Keeping train at 90% preserves training scale while making validation and test each large enough for stable metrics.
- A hard cutover avoids carrying old split semantics through the codebase.
- Rebuilding datasets and retraining the baseline costs time, but gives cleaner experiment comparisons afterward.
- The first implementation does not add configurable split ratios; fixed ratios keep the contract simple and reproducible.

## Rejected Alternatives

- Keeping the old 90/10 split and treating validation as final test: rejected because it mixes model selection and final evaluation.
- Adding a tiny test split without changing the train/validation split: rejected because the old validation contract remains muddled.
- Supporting both old and new split layouts: rejected because this is an internal research pipeline and a hard cutover is simpler.
