# Dataset Evaluation Reporting

## Design Goal

Create a reproducible dataset-level evaluation path for trained value-model checkpoints before using them in longer research loops. The first slice should answer whether a checkpoint predicts the existing side-to-move win-probability labels well on a known dataset split, without depending on interactive FEN entry, W&B summaries, or gameplay impressions.

## Current State

- Training optimizes binary cross entropy with logits against a side-to-move win-probability label.
- Training validation logs evaluate only a fixed number of validation batches per epoch, not necessarily the full validation file.
- The interactive `eval` command evaluates one FEN at a time and does not produce aggregate metrics.
- Processed `.chesseval` rows contain encoded tensors and `win_prob`; they do not preserve the original FEN or centipawn score.
- There is no dedicated test split yet, so validation reports must be treated as model-selection diagnostics rather than final unbiased performance claims.

## Agreed Model

Add a first-slice dataset evaluation command that loads a checkpoint, streams rows from a processed dataset, accumulates metrics, prints a concise console summary, and optionally writes a structured JSON report under ignored artifacts.

The command should be deterministic and explicit about what it measured:

```text
cpe eval-dataset MODEL_NAME
  --split validation
  --dataset PATH
  --rows N
  --full
  --batch BATCH_SIZE
  --seed SEED
  --device auto|cpu|cuda|mps
  --output PATH
```

Recommended defaults:

- split: `validation`
- dataset: resolved from the project data layout for the selected split
- rows: `1000000` deterministic prefix rows unless `--full` is set
- batch size: `4096`
- seed: `0`
- output: `artifacts/reports/MODEL_NAME.SPLIT.eval.json`

`--rows` and `--full` are mutually exclusive. When neither is provided, the
default is `--rows 1000000`. Prefix sampling is acceptable for the first slice
because the current preprocessing query randomizes row order before writing
train and validation files.

## Report Contract

The JSON report should be stable enough for later comparison scripts:

```json
{
  "schema_version": 1,
  "model": {
    "name": "model-name",
    "checkpoint_path": "artifacts/checkpoints/model-name-best.pth",
    "checkpoint_epoch": 70,
    "checkpoint_best_validation_loss": 0.6101640030449512
  },
  "data": {
    "split": "validation",
    "dataset_path": "data/processed/validation.chesseval",
    "dataset_rows": 15552523,
    "evaluated_rows": 1000000,
    "selection": "deterministic-prefix|full",
    "seed": 0
  },
  "run": {
    "device": "cuda",
    "batch_size": 4096,
    "torch_version": "2.9.1+cu128",
    "duration_seconds": 0.0
  },
  "metrics": {
    "bce_loss": 0.0,
    "brier_score": 0.0,
    "prob_mae": 0.0,
    "prob_rmse": 0.0,
    "cp_equivalent_mae": 0.0,
    "cp_equivalent_rmse": 0.0,
    "calibration_ece": 0.0,
    "baseline_bce_loss_0_5": 0.0,
    "baseline_brier_score_0_5": 0.0
  },
  "calibration_bins": [
    {
      "lower": 0.0,
      "upper": 0.1,
      "count": 0,
      "pred_mean": null,
      "target_mean": null,
      "abs_gap": null
    }
  ],
  "warnings": [
    "validation_split_is_model_selection_data"
  ]
}
```

## Metrics

- `bce_loss`: primary training-objective metric.
- `brier_score`: probability quality metric.
- `prob_mae` and `prob_rmse`: direct probability-space errors.
- `cp_equivalent_mae` and `cp_equivalent_rmse`: inverse-logistic centipawn-equivalent errors computed from clamped probabilities.
- `calibration_ece`: weighted expected calibration error over fixed probability bins.
- Baselines against constant `0.5` prediction should be included so a report can detect obvious regressions.

Loss alone is not enough because it does not reveal calibration quality, probability-scale error, or how much better the checkpoint is than a trivial baseline.

Metric aggregation must be row-weighted, not batch-weighted, so results do not
change when the last batch is smaller or the batch size is changed.

Metric constants:

- calibration uses 10 bins: `[0.0, 0.1)`, `[0.1, 0.2)`, ..., `[0.9, 1.0]`
- `p == 1.0` belongs to the last bin
- inverse logistic uses `cp = 400 * log10(p / (1 - p))`
- predictions and targets are clamped to `[1e-6, 1 - 1e-6]` before inverse logistic

## Boundaries And Invariants

- The model output is a raw logit for side-to-move win probability; evaluation must apply sigmoid before probability metrics.
- Evaluation must compare predictions to the processed dataset's side-to-move `win_prob` label, not White probability.
- Accumulators must stream batches and avoid storing all predictions for large splits.
- Output ordering in JSON must be deterministic.
- Sampling must be deterministic and recorded in the report.
- Empty calibration bins must be represented explicitly with `null` aggregate values.
- Probability-to-centipawn conversion must clamp probabilities away from `0` and `1`.
- `checkpoint_epoch` must be reported as raw checkpoint metadata, not normalized across checkpoint types.
- Validation reports must warn that the split is used for model selection until a separate test split exists.
- Add `artifacts/reports/` to the runtime artifact layout and keep generated JSON reports ignored by Git.
- If the evaluator is implemented as a new top-level Python module, register it in `pyproject.toml`.

## Failure Modes

- Missing checkpoint or dataset path should fail with a clear message.
- A requested row count larger than the dataset should evaluate only available rows and record the actual count.
- Passing both `--rows` and `--full` should fail clearly before loading the model.
- Device selection should fail clearly when the requested device is unavailable.
- A checkpoint with incompatible model weights should fail rather than using a compatibility shim.
- Non-finite logits or metric values should be surfaced as an evaluation failure.

## Tradeoffs

- Start with dataset metrics instead of gameplay strength because they are cheaper, deterministic, and directly tied to the current supervised objective.
- Use the existing processed dataset format even though it lacks FEN and original centipawn scores. This keeps the first slice narrow.
- Keep W&B logging out of the first slice. Reports can be uploaded or compared later.
- Use validation first for immediate research feedback, but avoid presenting it as final model quality.

## Rejected Alternatives

- Relying on W&B validation loss only: rejected because training validation is sampled and does not expose calibration or probability-scale metrics.
- Making `battle` the primary evaluator: rejected for the first slice because it is slower, noisier, and measures a different layer of the stack.
- Regenerating the entire dataset with richer metadata before any evaluation command exists: rejected because it blocks immediate checkpoint assessment.

## Remaining Open Issues

- Add a true `test.chesseval` split for final model comparison.
- Decide whether later reports should include W&B run identifiers and optional W&B upload.
- Add position-suite or search-oriented evaluation once the dataset evaluator is stable.
- Decide whether deterministic prefix sampling is sufficient or whether seeded random index sampling should be the default for large files.
