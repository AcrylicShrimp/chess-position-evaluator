# Experiment: material label calibration and dataset geometry

## Goal

Measure whether the fixed material-score prior used by the parallel CNN attention fusion experiment matches the processed dataset labels, and check whether processed `.chesseval` rows contain invalid board geometry.

This experiment is meant to decide whether another material-prior model experiment is safe, or whether dataset source/cleaning work must happen first.

## Hypothesis Or Expectation

The original expectation was that raw material difference would roughly track the dataset's side-to-move win-probability labels. After the previous training run showed poor calibration and large gradient-norm tails, the revised expectation was that either the fixed prior scale is miscalibrated, the dataset contains outlier rows, or both.

## Key Characteristics

- No model checkpoint is loaded.
- The analysis reads processed `.chesseval` rows through the same dataset reader used for training.
- Material difference is computed from side-to-move-relative piece channels.
- Fixed material prior uses `material_diff * log(10) / 4`.
- Reports include all-row metrics, valid-geometry metrics, invalid-geometry metrics, material-diff buckets, absolute-material buckets, side-to-move buckets, and prior-probability calibration bins.
- Invalid geometry checks include excessive side piece totals, excessive pawns, invalid king counts, impossible material magnitude, and overlapping piece occupancy.

## Verdict Rationale

This is informational because it measures dataset and prior behavior rather than testing a pass/fail model change.

The evidence is strong enough to change project priority, but it should not be overstated. The processed datasets contain rare but extreme invalid board geometry. The invalid rate is about `0.04%`, and the outliers can produce material differences above `500`, which is large enough to corrupt fixed-prior logits and plausibly contribute to isolated gradient spikes. This rate is probably too low to explain the whole model-quality gap by itself.

The fixed material prior is also miscalibrated even after invalid rows are excluded. Valid-only prior BCE stays around `0.714`, which is worse than the constant `0.5` baseline BCE around `0.693`. This means the fixed material prior is not just noisy; under the current label distribution and scaling, it is actively worse than predicting an even game.

The more likely interpretation is two separate issues:

- Data cleaning still matters because nonstandard rows create extreme and misleading training examples.
- Material difference is still likely meaningful, but the fixed prior scale `log(10) / 4` is too aggressive or otherwise mismatched to this label distribution.
- After strict filtering and dataset regeneration, geometry contamination is resolved in analyzed splits, but fixed material prior BCE remains worse than the 0.5 baseline.

## Issues And Anomalies

- Processed rows include impossible piece geometry, including material ranges such as `-521..528`.
- A manually inspected validation smoke row had dozens of pieces in one piece plane, such as many queens on one side.
- `Board::from_str` rejects many bad rows already, but it is not a sufficient standard-position legality gate for rows with parse-valid but nonstandard piece counts.
- Source diagnosis found about `1.0%` rejected source rows in each 100k split sample, mostly `invalid_fen`. These mostly did not enter processed datasets under the old path because invalid FENs were already skipped.
- Regenerated strict-filtered datasets have no analyzer-detected invalid geometry in train 1M, validation full, or test full. Extreme material positions are now rejected by preprocess policy.
- Train full analysis was not run because the split has `139969662` rows; a deterministic 1M prefix sample was used. The processed dataset was randomized during DuckDB staging, so the prefix sample should still be representative enough for this diagnostic pass, but it is not a full train sweep.

## Follow-Up

- Treat strict dataset legality filtering and regeneration as the next highest-priority project task.
- Reconsider fixed material-prior design; clean-data calibration still shows the current fixed scale is worse than a 0.5 baseline.
- Treat checkpoints trained before strict filtering as historical baselines until a clean-data baseline is trained.
- Train a clean-data baseline before drawing new architecture conclusions.
