## Goal

Test whether dynamic edge-gated board attention underperformed because the prior architecture had only one shallow global attention layer.

This experiment should inform whether adding attention depth plus a small per-square channel FFN makes edge-gated attention a viable direction, or whether the current edge-gate mechanism remains inferior even with more capacity.

## Hypothesis Or Expectation

The one-layer edge-gate model learned non-trivial relation-dependent logits, but did not beat static relation-bias attention on full validation/test BCE and worsened ECE.

If edge-gated attention was too shallow, a three-layer edge-gated stack with FFN should improve full validation and test BCE over the one-layer edge-gate run.

## Key Characteristics

- Architecture variant: `ValueOnlyModel` with `CHANNELS=256`.
- Trunk: six `GhostShuffleBlock` modules with coordinate attention.
- Attention insertion point: after trunk block 3, unchanged.
- Attention stack: `3` `BoardAttentionBlock` layers.
- Attention layout per layer: `4` heads, `16` dimensions per head, total projection dimension `64`.
- Edge gate: same relation/distance dynamic edge gate as the prior one-layer edge-gate run.
- FFN per stack layer: per-square channel FFN, `256 -> 64 -> 256`, with `BatchNorm2d(64)` and `Hardswish`.
- Static relation/distance logit bias: not used.
- Expected `ValueOnlyModel` parameters: `457,685`.
- This is not parameter-matched against the one-layer edge-gate run.
- Design doc: `docs/designs/2026-05-26.01-stacked-edge-gated-attention-ffn.md`.

## Verdict Rationale

Verdict is `Inconclusive` because the planned 100-epoch recipe was intentionally stopped during epoch 55 after severe post-restart instability. The best checkpoint was from epoch 30, after about `62,914,560` training samples, or roughly `44.95%` of the train split before accounting for shuffle/order details.

If the epoch-30 checkpoint is scored mechanically against the pre-registered full-evaluation thresholds, it does not pass: full test BCE `0.6083701942476437` does not beat the one-layer edge-gate test BCE `0.6077498676321618`, and full validation BCE `0.608331389700455` does not beat `0.6076844683747489`.

That result is expected for an early checkpoint and should not be treated as a clean architecture failure. The more important finding is that the deeper attention stack learned faster through epoch 30, then the matched prior `lr=0.0005` warm-restart recipe destabilized it sharply. This run is therefore evidence for scheduler retuning before drawing a model-capacity conclusion.

## Issues And Anomalies

- This experiment is not parameter-matched; it intentionally increases `ValueOnlyModel` from `223,573` to `457,685` parameters.
- No smoke OOM was observed.
- Validation became severely unstable after the warm restart following epoch 30. Sampled validation moved from best `0.6078` at epoch 30 to `0.6225` at epoch 31, `0.6290` at epoch 35, `0.6377` at epoch 48, and `0.6241` at epoch 50.
- The instability suggests the matched prior max learning rate `0.0005` and warm-restart schedule are too aggressive for the deeper attention stack.
- Training was manually stopped during epoch 55 after the restart instability persisted.
- The best checkpoint was early at epoch 30, so it had seen only about `62,914,560` configured training samples out of `139,969,662` train rows.
- Final W&B summary recorded `train/grad_norm=5043.979`. The trainer logs the pre-clipping norm returned by `clip_grad_norm_`, so this is an instability diagnostic rather than the effective post-clipping update magnitude.
- Full validation BCE `0.608331389700455` was worse than the sampled best validation loss `0.6077890051346199`, so the sampled validation checkpoint signal was optimistic.

## Follow-Up

- Do not continue this exact `lr=0.0005`, `T_0=10`, `T_mult=2` recipe for the deeper attention stack.
- Run a scheduler retune experiment for the same architecture before changing the model again.
- Prefer testing a lower max LR such as `0.0003` and/or a non-restarting cosine decay, because the main failure appeared at warm restart rather than during initial descent.
- Add progress output or worker configurability to full dataset evaluation if repeated full evals remain slow.
