## Goal

Create a standardized quality/size/speed comparison across the existing saved
checkpoints that are most relevant to the small, fast, sufficiently strong model
objective.

This experiment is meant to decide which already-trained candidates are on the
practical Pareto frontier before spending more training budget.

## Hypothesis Or Expectation

The current best `stacked-edge-gate-ffn` checkpoint should remain the strongest
quality point, but faster candidates may expose a useful quality/speed tradeoff
that is not visible from the mixed timing evidence in earlier logs.

## Key Characteristics

- Benchmark-only experiment; no training.
- Uses existing best checkpoints and existing full validation/test JSON reports.
- Measures synthetic forward latency with one standardized command, batch size,
  dtype, device, compile mode, warmup count, and iteration count.
- Compares quality, parameter count, forward latency, throughput, and peak CUDA
  memory in one JSON report.

Speed-sorted observations:

- Fastest checkpoint: `no-attention`, `10.0501 ms`, but test BCE is weak at
  `0.6088827232878065`.
- Fastest non-trivial attention candidate: `funnel attn6 lr5e4`,
  `10.4018 ms`, with test BCE `0.602155957334376`.
- Best-quality checkpoint: `stacked lr7p5e4`, `12.3925 ms`, with test BCE
  `0.5997561140142528`.
- `depth-refresh lr7p5e4` is not attractive under this benchmark: it is slower
  than `stacked lr7p5e4` and worse on test BCE, although it uses less peak CUDA
  memory.
- `parallel q+k no-material` is larger, slower, and worse on BCE than
  `stacked lr7p5e4`; it is not on the practical frontier here.
- `parallel k-only no-material` is marginally faster than `stacked lr7p5e4`
  (`12.3398 ms` vs `12.3925 ms`) but loses `0.0026176493038057` test BCE and
  has more parameters, so the speed gain is too small to justify the quality
  drop.

## Verdict Rationale

Informational. This benchmark does not have a pass/fail threshold; it creates a
standardized Pareto snapshot for deciding the next experiment.

The main result is sharper than expected: the current best `stacked lr7p5e4`
checkpoint is not only the best quality point, it is also competitive on latency.
Most stronger-looking alternatives are dominated or nearly dominated under this
measurement. The only clearly faster attention candidate is the older
`funnel attn6 lr5e4`, but it gives up about `0.002400` test BCE versus the best
stacked checkpoint.

## Issues And Anomalies

- `rg --files artifacts/checkpoints artifacts/reports` hides ignored runtime
  artifacts; `find artifacts -maxdepth 3 -type f` confirmed the local
  checkpoints and reports are present.
- `max-autotune` emits verbose kernel-choice output while compiling. This is
  expected and was not treated as a failure.
- The measured forward benchmark uses synthetic random board tensors. It is
  appropriate for model execution speed, not for data-loading or end-to-end
  training throughput.
- After this run, the CLI was updated to default `TORCHINDUCTOR_CACHE_DIR` to
  `artifacts/cache/torchinductor` when the variable is not already set.

## Follow-Up

- Prefer `stacked-edge-gate-ffn` as the active quality/speed reference.
- For a faster model, the most credible existing branch is not parallel fusion;
  it is a retuned/narrowed funnel-style model, because the old funnel is much
  faster but quality-limited.
- If optimizing small/fast next, run a controlled narrower stacked or funnel
  experiment rather than more parallel-fusion work.
- If a latency decision depends on sub-1% differences, rerun this benchmark with
  more iterations before treating the ordering as stable.
