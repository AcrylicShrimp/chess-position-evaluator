## Goal

Train a no-material-input parallel CNN/attention fusion variant on the cleaned dataset.

This experiment isolates whether explicit material-score-derived inputs in `MaterialFeatureValueHead` are helping or hurting the parallel-fusion model. The trunk, dataset, and training recipe should match the previous clean `parallel-cnn-attn-fuse` material-feature run as closely as possible.

## Hypothesis Or Expectation

If explicit material features are causing overconfident or brittle behavior in the parallel model, removing them should improve full validation/test BCE or calibration versus the previous material-feature parallel run.

If the no-material variant is equal or worse, then the parallel-fusion gap is probably not explained primarily by explicit material-score inputs. In that case, the stronger next direction should move away from this parallel branch design or use cheaper diagnostics before another full parallel-fusion run.

## Key Characteristics

- Model variant: `parallel-cnn-attn-fuse-no-material`.
- Shared encoder: `GhostShuffleBlock x3`.
- Local branch: `GhostShuffleBlock x3`.
- Global branch: `BoardAttentionStack x3`, each with 4 heads, 16 head dim, and FFN hidden size 64.
- Fusion: concat local/global features followed by `Conv2d 1x1 512 -> 256`, BatchNorm, Hardswish.
- Value head: `ResidualValueHead`.
- Explicit material-score input: removed.
- Additive material prior: not used.
- Dataset: regenerated strict-filtered processed data under `data/processed`.
- Scheduler recipe: warmup-cosine, 5 warmup epochs, no warm restart.

## Verdict Rationale

Pass for the specific material-feature ablation question.

The full test BCE `0.601583460035304` is below the pre-registered clear material-removal win threshold `< 0.6023`, and it improves over the previous material-feature parallel run by about `0.00118` BCE on both full validation and full test. Calibration also improved substantially: full test ECE dropped from `0.0012650032161858132` to `0.0005251688410744738`.

This does not make the parallel-fusion architecture the current best model. The cleaned-data stacked edge-gate FFN still has better full validation/test BCE by about `0.00065`, despite worse ECE. The useful conclusion is narrower: explicit material-score-derived value-head inputs were hurting or at least constraining this parallel-fusion family, and removing them materially improved the run.

## Issues And Anomalies

- The source is not committed before the run; this is recorded explicitly in setup.
- This experiment is stacked on top of the uncommitted aligned-add implementation/logging changes.
- The first smoke attempt used `epochs=1` with `warmup_epochs=5`, which violates the scheduler contract. The corrected smoke preserved `warmup_epochs=5` and reduced steps instead.
- At epoch 10, the no-material variant was materially behind the previous material-feature parallel run. By epoch 20 it recovered to the same performance class, so the early negative signal is not decisive.
- By epoch 30, the no-material variant moved ahead of the previous material-feature parallel run by about `0.0012` sampled validation loss. This is the first meaningful positive signal for removing explicit material-score inputs.
- At epoch 40, no-material is essentially tied with the previous concat material-feature run while still showing larger sampled-validation swings. The final full evaluation is required to decide whether the swings are noise, calibration effects, or a true head-input difference.
- At epoch 50, no-material is slightly ahead of the previous material-feature concat run by best sampled validation loss. This supports continuing the full run rather than stopping on the noisy validation swings.
- At epoch 60, no-material has reached the previous material-feature epoch-69 best about ten epochs earlier. This strengthens the hypothesis that explicit material features may have been acting as a brittle shortcut rather than a useful input for this parallel trunk.
- At epoch 70, no-material essentially reached the previous material-feature epoch-89 best. This is now a strong sampled-validation signal that removing explicit material features may improve the parallel trunk's training dynamics, even if final full BCE still needs confirmation.
- At epoch 80, no-material entered the pre-registered clear material-removal win band. One plausible mechanism is that explicit material features acted as a convenient shortcut for the head, reducing pressure on the trunk to learn board-pattern structure directly.
- The final checkpoint was selected at epoch 100, so this run kept improving all the way to the schedule end. A longer or differently decayed schedule may still be worth testing later.
- Full evaluation confirms the sampled-validation signal. The model is better than the material-feature parallel baselines but still worse than the simpler stacked edge-gate FFN by full BCE.

## Follow-Up

- Treat explicit material-score-derived head inputs as suspect for future parallel-fusion variants.
- Do not use this parallel-fusion family as the mainline architecture yet; cleaned-data stacked edge-gate FFN remains the stronger BCE baseline.
- Use the no-material result when designing the next fusion experiment: any reintroduced material information should be an input feature the trunk can learn to use, not a hard shortcut or direct value-head residual unless separately justified.
