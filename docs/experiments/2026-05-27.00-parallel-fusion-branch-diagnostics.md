## Goal

Diagnose why the clean-data `parallel-cnn-attn-fuse` material-features run underperformed the clean-data stacked edge-gate FFN run.

This experiment is meant to inform whether the weak result is more likely caused by the local branch, attention branch, material feature path, fusion interface, or a broader optimization/generalization issue.

## Hypothesis Or Expectation

Exploratory. The main expectation was that intervention metrics and bucketed loss deltas would expose whether the parallel model fails broadly or only on a specific subset such as high-material-imbalance positions.

## Key Characteristics

- Diagnostic target: clean-data `parallel-cnn-attn-fuse` with `MaterialFeatureValueHead`.
- Baseline target: clean-data stacked edge-gate FFN.
- Evaluation split: clean validation split, deterministic prefix of `200000` rows.
- Diagnostic modes:
  - `parallel_full`
  - `parallel_local_only`
  - `parallel_global_only`
  - `parallel_no_material`
  - `parallel_material_only`
  - `baseline_full`
- Branch ablations are post-training interventions, not independently trained branch-quality measurements.

## Verdict Rationale

Informational. This diagnostic did not define a pass/fail threshold. It provides evidence about the failure shape of the parallel architecture.

The main finding is that the parallel model is not merely worse in aggregate. It is especially worse on confident-label buckets and moderate-to-large material-imbalance buckets.

The source-position trace refines the earlier material-shortcut interpretation. Material/label sign conflicts exist and include the worst two rows, but they are not the whole story: only `7 / 20` traced worst rows had material/label sign conflicts. The traced rows are heavily enriched for mate-clamped labels, neutral-cp positions, sparse endgames, promotions, and tactical-looking positions where small board details dominate static material.

The more defensible conclusion is that the parallel fusion model is brittle on sharp tactical/endgame cases and often produces overconfident probabilities. Explicit material features may contribute to that brittleness, but the trace does not support claiming material over-reliance as the only or dominant failure mode.

The `no_material` and `material_only` interventions should not be read as clean ablations because the head was trained with all inputs present. They still indicate that the material feature path is highly entangled with the learned representation. Removing material features after training damages the checkpoint substantially.

The local and global branch representations are nearly orthogonal by flattened cosine similarity, and the attention/global branch has larger activation scale than the local branch. That does not prove the cause of the underperformance, but it makes the 1x1 fusion boundary a plausible place to inspect next.

## Issues And Anomalies

- The diagnostic uses processed dataset indices, not source FENs. Top problematic rows are identified by `dataset_index` only.
- Branch ablations zero trained inputs after training, so they measure sensitivity, not independent branch quality.
- The diagnostic ran on a deterministic validation prefix of `200000` rows, not the full split. The prefix reproduces the full-eval gap direction and approximate magnitude, but full diagnostic buckets were not computed.
- The first source-trace attempt produced mismatched labels/material values because it did not include `python-chess` `STATUS_VALID`; that attempt was discarded and replaced with a validated trace.

## Follow-Up

- Source-position lookup now works for traced diagnostic rows when the matching staging DuckDB file is available.
- Inspect fusion-layer weights, branch normalization, and whether local/global scales should be explicitly normalized before fusion.
- Run a controlled model without the material feature path or with a weaker/gated material feature path if we want to isolate material over-reliance.
- Treat stacked edge-gate FFN as the stronger current baseline until parallel fusion has a controlled improvement path.
