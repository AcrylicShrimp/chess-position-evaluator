## Goal

Train the aligned-add local/global parallel fusion variant for a full 100-epoch run on the cleaned dataset.

This experiment is meant to determine whether explicit per-branch normalization and projection before additive fusion improves on the previous `parallel-cnn-attn-fuse` material-feature run while keeping the training recipe and branch bodies fixed.

## Hypothesis Or Expectation

The aligned-add fusion contract should make the local CNN branch and global attention branch interact in a more coherent shared feature space than naive concat fusion. Expected positive signs are:

- Comparable or better early validation trajectory than the previous `parallel-cnn-attn-fuse` material-feature run.
- Better final full validation/test BCE than the previous `parallel-cnn-attn-fuse` material-feature run.
- No worse stability than the previous parallel run, especially in gradient-norm tails.
- Similar throughput and only a small parameter increase.

## Key Characteristics

- Model variant: `parallel-cnn-attn-aligned-add`.
- Shared encoder: `GhostShuffleBlock x3`.
- Local branch: `GhostShuffleBlock x3`.
- Global branch: `BoardAttentionStack x3`, each with 4 heads, 16 head dim, and FFN hidden size 64.
- Fusion: per-branch BatchNorm, per-branch `Conv2d 1x1 256 -> 256`, elementwise add, final BatchNorm, Hardswish.
- Output contract: `value_logit = value_head(fused_features, material_features)`.
- Value head: `MaterialFeatureValueHead`, matching the previous material-feature parallel run.
- No additive material prior is added to the output logit.
- Dataset: regenerated strict-filtered processed data under `data/processed`.
- Scheduler recipe: warmup-cosine, 5 warmup epochs, no warm restart.

## Verdict Rationale

Inconclusive. The aligned-add fusion contract did not produce a material improvement over the previous concat-fusion material-feature run. Full validation and test metrics are slightly worse across BCE, ECE, and CP MAE, but the BCE deltas are only about `0.00016..0.00019`, so the practical reading is same performance class rather than a strong regression.

The result lands in the `Neutral` full-test BCE band against the previous parallel concat material-feature run: `0.6027627174346522..0.6032`. The experiment therefore does not support replacing the simpler concat-fusion implementation with aligned-add, unless a later diagnostic shows a specific stability or interpretability reason to prefer aligned-add.

## Issues And Anomalies

- The source was not committed before the run; this is recorded explicitly in setup.
- An initial attempt was incorrectly started in offline W&B mode because the shell did not automatically source `.env`. The user interrupted the run, `.env` was confirmed to contain W&B credentials, and the run was restarted after explicitly sourcing `.env`.
- The aborted offline attempt reached checkpoint metadata `epoch=2` with sampled best validation loss `0.6443404637131036`. Its partial checkpoints were removed before the clean online restart. The offline W&B directory was left at `wandb/offline-run-20260527_130604-9wq5u0m0`.
- The first online restart wrote high-volume tqdm output to the Codex session. It was intentionally stopped at checkpoint metadata `epoch=3` with sampled best validation loss `0.63746697937741`, then restarted with stdout/stderr redirected to `artifacts/reports/ghost-ca-r4-256ch-parallel-cnn3-attn3ffn64-alignedadd-materialfeatures-warmupcosine5-lr5e4-100e.train.log`.
- The first online restart W&B run ID was `mnfsfstf`; the file-logged clean run uses W&B run ID `77js0au3`.
- Through epoch 50, sampled validation differences versus the previous `parallel-cnn-attn-fuse` material-feature run are too small to call a material improvement or regression. Treat the current trajectory as effectively the same performance class until full validation/test metrics say otherwise.
- Final full validation/test evaluation confirmed the same-class interpretation: aligned-add was consistently but only slightly worse than concat fusion.
- The comparison reference table was corrected at closeout to use the cleaned-data stacked edge-gate FFN retrain as the stacked baseline. This does not change the primary aligned-add versus concat-fusion conclusion.

## Follow-Up

- Do not promote `parallel-cnn-attn-aligned-add` as the primary parallel-fusion path based on this run.
- Next experiment should avoid spending another full run on small fusion arithmetic changes alone.
- Better next candidates are: isolate material features on the stronger stacked edge-gate FFN path, or design a cheaper branch-diagnostic probe before testing another parallel-fusion architecture.
