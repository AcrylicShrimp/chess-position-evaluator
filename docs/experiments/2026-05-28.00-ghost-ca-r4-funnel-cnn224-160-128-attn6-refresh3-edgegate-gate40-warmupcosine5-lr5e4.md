## Goal

Run a gated depth-preserving refresh probe for the funnel attention family.

This experiment tests whether adding CNN refresh blocks after pairs of narrow attention blocks can preserve the previous six-attention funnel's relation-mixing depth while reducing the final-stage spectral concentration observed in that run.

The decision this experiment informs is whether the funnel-refresh direction deserves a full 100-epoch run or should be deprioritized in favor of the cleaned stacked baseline family.

## Hypothesis Or Expectation

The expected positive signal is that preserving six attention blocks while adding three refresh blocks improves sampled validation loss versus the previous six-attention funnel by epoch 40 and avoids the severe final-attention spectrum concentration.

The main risk is that the prior interleaved rank improvement came from added CNN capacity rather than useful task signal, or that `C128` remains too narrow regardless of refresh.

## Key Characteristics

- Model variant: `funnel-cnn224-160-128-attn6-refresh3-edgegate`.
- Architecture: `24 -> 224`, two `C224` Ghost blocks, `224 -> 160`, two `C160` Ghost blocks, `160 -> 128`, two `C128` Ghost blocks, then `[C128 q+k edge-gated attention block -> C128 q+k edge-gated attention block -> C128 Ghost refresh] x3`.
- Attention: `4` heads, `16` head dim, FFN hidden size `64`.
- Value head: material-difference scalar semantics matching the cleaned stacked baseline and previous funnel experiments.
- Distinct from the previous six-attention funnel: adds three CNN refresh blocks while preserving six attention blocks.
- Distinct from the previous interleaved funnel: preserves six attention blocks instead of reducing attention depth to three.
- Design doc: `docs/designs/2026-05-28.00-funnel-depth-preserving-refresh.md`.

- Rank diagnostics improved substantially versus the previous final-attention concentration, but better rank shape did not translate into better BCE.
- Grad norm in the fresh 100e run was normal: max `2.13983`, p99 `0.472445`, and zero steps above clip threshold `5`.
- The gate40 first-epoch grad spike did not reproduce in the fresh 100e run, so it should be treated as a run-specific anomaly rather than proven architectural instability.

## Verdict Rationale

Pass for the pre-registered gate. The epoch-40 sampled validation loss was `0.6062`, better than the previous six-attention funnel's `0.6072` and within the strong extension threshold `<= 0.6067`. Rank analysis also avoided the previous final-attention concentration failure: final attention `rank>10%` improved from `21/128` to `48/128`, and the final refresh block measured `86/128`.

The full 100-epoch result failed the architecture promotion criterion. Although the gate passed and the full run's rank diagnostics improved over the previous six-attention funnel, full validation and test BCE regressed versus both primary references.

The gate40 grad-norm spike is a real anomaly but appears first-epoch localized and run-specific. A fresh 100-epoch run of the same architecture did not reproduce the spike cluster.

## Issues And Anomalies

- Initial dataset row-count probe failed from the `python/` working directory because relative paths resolved against the wrong root. Re-run from repository root with `PYTHONPATH=python` succeeded and matched existing row counts.
- The 40-epoch gate was run from uncommitted implementation changes. The implementation and initial gate log were committed before starting the full 100-epoch run.
- Do not use `--resume` from the 40-epoch gate for the full 100-epoch comparison. The checkpointed scheduler state preserves the original `total_epochs=40` horizon.
- W&B grad-norm history shows a severe gate40 first-epoch spike cluster: max `114.117`, p99 `33.365`, and `192` first-epoch steps above the configured clip threshold `5`. The logged value is the pre-clipping total norm returned by `torch.nn.utils.clip_grad_norm_`, so optimizer steps were clipped, but the signal indicates unusually unstable early gradients.
- The same depth-refresh architecture in the fresh 100e run did not reproduce the spike cluster: max `2.13983`, p99 `0.472445`, and zero steps above `5`.
- Sampled validation overstated full-split quality for the 100e best checkpoint. The checkpoint's sampled validation was `0.6028233211414487`, but full validation BCE was `0.6031740756086265`.

## Follow-Up

- Commit the completed experiment log update.
