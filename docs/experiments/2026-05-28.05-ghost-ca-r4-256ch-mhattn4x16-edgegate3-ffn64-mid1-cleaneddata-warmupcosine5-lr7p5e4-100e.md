## Goal

Run the strongest cleaned-data stacked edge-gate FFN architecture with `lr=0.00075` for 100 epochs.

This experiment tests whether the `lr=0.00075` improvement observed in the depth-refresh family is an architecture-specific result or a broader optimizer setting that also improves the previous stacked edge-gate FFN baseline.

## Hypothesis Or Expectation

The stacked edge-gate FFN architecture should remain stable at `lr=0.00075` and improve over its cleaned-data `lr=0.0005` reference.

Expected positive evidence:

- no divergence or epoch-20s instability comparable to the previous `lr=0.001` gate failure in the depth-refresh family
- sampled validation reaches or beats the cleaned stacked `lr=0.0005` trajectory by late training
- full validation BCE beats `0.6008478613148978`
- full test BCE beats `0.6009210722625944`

Strong evidence:

- full validation BCE beats the current depth-refresh best `0.600658`
- full test BCE beats the current depth-refresh best `0.600767`

## Key Characteristics

- Model variant: `stacked-edge-gate-ffn`.
- Architecture is unchanged from the cleaned-data stacked edge-gate FFN reference.
- Training recipe follows the current warmup-cosine, no-restart recipe.
- Base learning rate is raised from `0.0005` to `0.00075`.
- This run is intended as an optimizer-level comparison against the current best depth-refresh result.

## Verdict Rationale

Pass. The run satisfies both the expected positive criteria and the strong evidence criteria.

This is the new best recorded full validation/test result in the current experiment log set. It beats the previous depth-refresh `lr=0.00075` full run by about `0.000951` validation BCE and `0.001011` test BCE. It also beats the same-architecture cleaned stacked `lr=0.0005` reference by about `0.001141` validation BCE and `0.001165` test BCE.

The main conclusion is that the earlier stacked edge-gate FFN result was learning-rate limited. Raising the base LR from `0.0005` to `0.00075` did not introduce instability; instead it improved the whole late-training trajectory and produced better calibration.

## Issues And Anomalies

- The run plateaued near the end: sampled validation improved sharply through epoch 90 and then only marginally through epoch 100.
- W&B API access required loading `WANDB_API_KEY` from `.env` in the local process; the key was not printed.
- The validation eval report includes `validation_split_is_model_selection_data`, as expected for evaluating the best checkpoint on the validation split.

## Follow-Up

- Promote `stacked-edge-gate-ffn` with `lr=0.00075` as the current best baseline.
- Treat LR tuning as a first-class axis in future architecture comparisons.
- Consider a small LR sweep around this point, such as `0.000625`, `0.00075`, and `0.000875`, before making a major architecture change.
- Re-evaluate whether recent architectural variants were fairly tuned before rejecting them.
